## 3. 实际解决了什么业务问题

### 3.1 解决了“素材接收混乱”的问题

原始 Telegram 媒体组容易退化成一串文本链接，无法稳定识别“主视频”“正文图”“遮罩图”“参考图”。

现在系统把素材收件标准化为批次清单和稳定文件名，能明确：

- 哪个视频才是主素材
- 哪些图片进入正文
- 哪些图片只是遮罩或参考
- 一个会话是单任务还是多任务

### 3.2 解决了“大文件下载不稳”的问题

对大视频不再粗暴全量下载，而是支持：

- `needs_project_download` 标记
- 断点续传
- 指数退避重试
- 下载进度、速度、ETA 报告
- 慢速网络告警
- 失败后的结构化恢复建议

这直接解决了 Telegram 大文件下载超时、Bot API 不稳定和本地服务重启风险。

### 3.3 解决了“视频处理和发布链路割裂”的问题

以前视频处理、文案、预览、上传、建稿、发布确认是分散动作。

现在已经串成一条单任务链路：

- 接任务
- 校验参数
- 处理视频
- 生成文案
- 生成预览
- 上传媒体
- 建稿/发布
- 浏览器确认和截图取证

### 3.4 解决了“人工发布不透明、失败难复盘”的问题

系统现在不仅能跑，还能留证：

- 每个任务都有 `stage.log`
- 有 `preview/article_preview.png`
- 有浏览器过程截图
- 有失败证据 JSON
- 能记录页面类型、命中元素、候选行、DOM 摘要和网络记录

这让“为什么失败”“失败停在哪一步”从不可诊断变成可诊断。

## 5. 优化前投入情况（保守测算）

仓库内没有历史财务台账，因此这里采用流程拆解法做保守测算。

以 1 条“清洗视频并发布”的标准任务为例，优化前通常至少涉及 2 个角色：

- 角色 A：收件、挑素材、下载、去水印、加品牌水印、导出
- 角色 B：写标题/正文/关键词、进后台上传、建稿、检查落点

按当前流程节点倒推，传统人工方式的典型投入大致是：

| 环节 | 人工耗时（保守） |
| --- | --- |
| 收件、整理素材、确认主视频 | 5-10 分钟 |
| 视频处理、去水印、加品牌水印 | 15-25 分钟 |
| 文案、标题、关键词、配图排版 | 10-15 分钟 |
| 后台上传、建稿、发布确认 | 10-15 分钟 |
| **合计** | **40-65 分钟/单** |

按角色看，通常是：

- 至少 2 人参与
- 单单总历时约 40-65 分钟
- 人工净投入约 35-50 分钟/单

## 6. 优化后实现方式：现在 AI 如何替代或分担

现在的实现方式不是“一个大模型包打天下”，而是分层替代：

- `shipinfabu` 负责接任务、澄清、建单、轮询和汇报
- `clean-publish-intake-clarify` 负责素材分批和阻塞项澄清
- `media-pipeline-service` 负责视频处理、文案、预览和发布执行
- `publish_browser` 负责真实页面确认、存草稿/发布补写和截图留证
- `publishhub-browser-ops` 负责人工接管前的只读巡检

也就是说，AI/自动化现在替代的是：

- 素材标准化和任务参数整理
- 视频处理流水线编排
- 文案生成和质量规则执行
- 平台上传、建稿、发布动作
- 页面确认、日志记录和失败取证

人只保留两类动作：

- 任务开始前确认真正阻塞的信息，例如原水印位置或主素材选择
- 发布异常时做人工接管和复核

## 7. 节省结果（保守测算）

继续按单任务保守测算：

- 优化前：40-65 分钟/单，至少 2 个角色参与，人工净投入 35-50 分钟
- 优化后：AI 自动跑主链，人工只做澄清和异常兜底，人工净投入通常降到 5-10 分钟

因此可以保守给出以下结果：

- 单任务人工净投入下降约 71%-86%
- 参与角色从“至少 2 个角色协作”降到“1 人值守 + 系统自动执行”
- 系统主链已被压缩到分钟级，典型样本显示 1-2 分钟可推进到建稿/浏览器确认阶段

如果按 100 单/月粗算：

- 优化前人工净投入约 58-83 小时/月
- 优化后人工净投入约 8-17 小时/月
- 月度节省约 41-75 小时人工

如果按 160 小时/月折算，约等于：

- 节省 0.26-0.47 个全职人月

这是保守口径；如果原流程确实存在多人来回交接、重试和返工，实际节省会更高。
